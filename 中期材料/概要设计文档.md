# 作业OCR自动批改系统 — 概要设计文档

## 1. 引言

### 1.1 编写目的

本文档是"作业OCR自动批改系统"的概要设计文档，旨在从系统整体架构层面描述系统的设计方案，包括系统架构、模块划分、数据流、接口设计和数据库概要设计，为后续详细设计和开发提供指导。

### 1.2 项目背景

本系统面向教师用户，提供基于OCR识别的作业自动批改功能。教师拍照或扫描学生作业后，系统自动识别手写/印刷内容，与标准答案比对并给出评分，支持导出报告和历史记录查询。

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| OCR | 光学字符识别 (Optical Character Recognition) |
| PaddleOCR | 百度开源的OCR识别引擎 |
| 填空题 | fill_blank，需要文本匹配的题型 |
| 选择题 | multiple_choice，字母选项匹配的题型 |
| 计算题 | calculation，数值比较的题型 |

### 1.4 参考资料

- PaddleOCR 官方文档
- Flask 官方文档
- PyQt5 官方文档
- SQLAlchemy 官方文档

## 2. 总体设计

### 2.1 系统架构

本系统采用 **前后端分离的C/S架构**，前端为PyQt5桌面应用，后端为Flask REST API服务，两者通过HTTP协议通信，均运行在本地。

![概要设计文档](https://mp-2b1c41a4-fdd7-465e-af62-0e39538234ac.cdn.bspapp.com/ustc_gcsj/概要设计文档/2-1.png)

### 2.2 设计原则

1. **模块化设计**：各核心功能独立封装，降低耦合
2. **前后端分离**：前端负责展示和交互，后端负责业务逻辑和数据处理
3. **单一职责**：每个模块只负责一项核心功能
4. **可扩展性**：题型、导出格式等可方便扩展

### 2.3 运行环境

| 项目 | 说明 |
|------|------|
| 操作系统 | Windows / macOS / Linux |
| Python | >= 3.8 |
| 数据库 | SQLite 3 |
| 网络 | 本地回环 (127.0.0.1) |

## 5. 接口概要设计

### 5.1 REST API 接口列表

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/health` | 健康检查 |
| POST | `/api/upload` | 上传作业图片 |
| POST | `/api/preprocess/<file_id>` | 图像预处理 |
| POST | `/api/ocr/<file_id>` | OCR文字识别 |
| POST | `/api/parse` | 解析OCR结果 |
| POST | `/api/grade` | 完整批改流程（OCR+解析+评分） |
| POST | `/api/export/<file_id>` | 导出报告 |
| GET | `/api/history` | 查询批改历史 |
| GET | `/api/history/<id>` | 查询单条记录详情 |
| DELETE | `/api/history/<id>` | 删除批改记录 |
| GET | `/api/statistics` | 获取统计信息 |

### 5.2 核心内部接口

| 模块 | 接口 | 输入 | 输出 |
|------|------|------|------|
| OCR引擎 | `recognize(image)` | 图片路径或numpy数组 | `List[OcrResult]` |
| 预处理器 | `preprocess(image_path)` | 图片路径 | numpy数组 |
| 解析器 | `parse_answers(ocr_results)` | OCR结果列表 | `Dict[int, str]` |
| 批改引擎 | `grade_all(parsed, questions)` | 解析结果+题目列表 | `GradingReport` |
| 导出器 | `export_csv/html(results)` | 批改结果列表 | CSV/HTML字符串 |

## 6. 数据库概要设计

### 6.1 E-R图

![E-R图](https://mp-2b1c41a4-fdd7-465e-af62-0e39538234ac.cdn.bspapp.com/ustc_gcsj/概要设计文档/6-1-fix.png)

### 6.2 表结构概要

- **homeworks**：存储上传的作业图片信息
- **grading_records**：存储每次批改的汇总结果
- **question_results**：存储每道题的批改详情

### 6.3 关键关系

- Homework → GradingRecord：一对多（一张作业可多次批改）
- GradingRecord → QuestionResultRecord：一对多（一次批改包含多道题）

## 7. 安全性设计

| 方面 | 措施 |
|------|------|
| 网络安全 | 仅监听 127.0.0.1，不暴露到外网 |
| 文件上传 | 限制文件大小 (16MB)、文件类型白名单 |
| 文件存储 | 使用UUID重命名，防止路径遍历 |
| SQL注入 | 使用 SQLAlchemy ORM 参数化查询 |

---

## 8. 性能设计

| 方面 | 措施 |
|------|------|
| OCR引擎 | 懒加载单例模式，避免重复初始化 |
| 图像处理 | 异步线程处理，避免UI阻塞 |
| 数据库 | 分页查询，避免大量数据一次性加载 |
| 前端 | QThread后台执行批改任务 |
